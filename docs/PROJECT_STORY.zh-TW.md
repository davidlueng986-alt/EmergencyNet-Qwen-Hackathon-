## Elevator Pitch

以下每個英文對應版本均少於 200 words。報名表或影片只需選一個，不要把三段拼成一段。

### 1. 30 秒直白版

EmergencyNet 讓救援隊在流動網絡中斷時，仍可完成檢傷和回報。前線人員在 Android 平板輸入傷患資料，本機的 12 問規則會產生檢傷標籤，並找出容易漏看的延遲風險。系統把結果轉成十六進位文字，操作員貼到 Meshtastic，Heltec V4 再經 LoRa 網狀網絡接力。指揮站把多名傷患的資料合併，找出呼吸異常、燒傷、受困及大量 RED 傷患等群聚。Qwen 可覆核筆記、演練影像、整理局勢及草擬短訊，但不能降低風險、改標籤或自行發送。決定仍由人作出；軟件負責減少遺漏和整理資訊。

### 2. 評審版

建築物倒塌後，前線人員可能每名傷患只有一分鐘，網絡亦未必可用。EmergencyNet 提供一條可重複的流程：輸入觀察、離線執行 12 問、查看 BLACK／RED／YELLOW／GREEN 的具體原因，再用 Heltec V4 Meshtastic LoRa Mesh 傳送精簡紀錄。目前 Demo 坦白保留兩次人工交接：把 Field 產生的 Hex 貼到 Meshtastic，再把 Base 收到的 Hex 貼進 Dashboard。Base 還原傷患、建立 SITREP，並找出跨傷患模式。Qwen Cloud 處理適合語言模型的工作：多語筆記、演練影像、戰術建議、策略整理及警報草稿。程式限制它只能提出升級；救援人員必須接受，Agent 沒有改標籤工具，廣播亦只能由另一個人工控制核准。Base 已封裝供 Alibaba Cloud ECS 部署並使用 Model Studio API；正式提交前仍要補上真實運行證明。

### 3. 技術版

EmergencyNet 是離線優先的前線至指揮系統。確定性 Python 核心把結構化觀察轉成 12 個三態答案、八項隱藏風險、檢傷標籤、信心值、覆核提示及優先分數。12 個兩位元答案剛好佔三 bytes；每名傷患完整紀錄為 18 bytes。Hex 會令文字長度加倍，所以 Field 現時輸出獨立四人封包，而不是聲稱 12 人 Hex 可放入 Meshtastic。Base 的嚴格解碼、有限事故視窗、SITREP 及四個群聚偵測器不用 Qwen 也可運作。`qwen3.7-plus` 覆核多語筆記、演練影像及執行六工具 Agent；`qwen3.7-max` 根據即時快照提供結構化策略。模型不能修改傷患或核准發送。Base 已準備部署至 Alibaba Cloud ECS，並透過相容 OpenAI 的 API 呼叫 Model Studio；自動收發 Radio 仍屬下一階段。

## 專案故事

### 我們針對甚麼情況設計

想像建築物倒塌後的第一小時。流動網絡超載或完全中斷，一小隊前線人員卻找到數十名傷患。有人明顯危殆；也有人仍能說話和步行，但身上有稍後才惡化的警號。救援人員要同時評估、記錄、決定誰先撤離，還要把情況回報指揮站。

指揮站面對相反難題：收到多隊人的短訊，卻看不到全局。一宗呼吸問題可能只是個別個案；十分鐘內出現五宗，則可能與煙霧或共同危害有關。真正重要的資料，往往是多份報告合起來才看見的模式。

EmergencyNet 就是為了接通這兩個視角，而且不假設流動網絡一定存在。目標很實際：讓小隊少漏掉重要資料、用 Radio 傳送最少但有用的結構化紀錄，再讓指揮員較快看清整體情況。我們不會聲稱這個 Prototype 已拯救某個數目的生命，因為現時沒有臨床驗證及真實部署證據支持這種說法。

如果團隊有經核實的個人或社區經歷，可用自己的說話補在這裡：**[團隊提供的真實靈感來源]**。

### 為甚麼不是先做一個 Chatbot

最容易做的 Hackathon 版本，是把表格直接接到大型模型。但當網絡不穩、API Key 不存在、筆記含糊或模型答案飄忽時，它也會在最需要的時候失效。

因此，我們把工作清楚分開：

- 固定程式負責可重複的檢傷計算；
- Qwen 處理語言、演練影像及多名傷患的資訊整理；
- LoRa Mesh 搬運最少但足夠的紀錄；
- Base 尋找整宗事故的模式；
- 人負責接受變更及批准對外行動。

這種分工不是為了令架構圖看起來複雜，而是為了控制失敗方式。Qwen 斷線時，隊伍只會失去額外覆核和整理，不會失去 12 問、封包、Radio 接力、解碼、群聚偵測及 SITREP。模型提出錯誤建議時，人可以拒絕。Radio 封包損壞時，解碼器會拒收，而不是自行猜出一名傷患。

### 為甚麼是 12 問

人在壓力下不應靠記憶重建整份流程。EmergencyNet 因此把評估整理成 12 個答案：`Yes`、`No` 或 `Unknown`。

Q1–Q4 是 Prototype 內受 START／JumpSTART 啟發的即時分支：重新打開氣道後仍沒有呼吸、呼吸異常、摸不到橈動脈脈搏，以及不能服從簡單指令。Q5–Q12 則處理初時未必明顯的危險：長時間受困、鈍性腹部創傷、懷孕警號、意識改變、氣道燒傷線索、嚴重但無痛的傷勢、長者頭部受傷兼混亂，以及近距離爆炸暴露。

固定 12 問有三個實際原因：

1. **減少記憶負擔。**每名傷患都跟同一個短流程，不用由自由文字臨場猜測下一步。
2. **結果可追查。**BLACK 或 RED 都可追溯至某條問題和規則，而不是無法重現的模型分數。
3. **適合 Radio 封包。**每個三態答案用兩 bits，12 個答案剛好佔三 bytes。

「12」是本 Prototype 的工程選擇，不代表這套問題已是完整、全球通用的醫療 Protocol。當中的門檻和行動文字，仍需合資格的本地臨床及事故指揮人員審核。

### 為甚麼要把隱藏風險分開顯示

即時檢傷主要問：「現在誰最需要處理？」它未必完整回答：「誰目前看來穩定，但稍後可能惡化，或者需要不同資源？」例如，口鼻附近有煙灰的人此刻仍能說話，氣道卻可能之後腫脹；長時間受困的人被救出時看似穩定，解除壓力後仍有風險；長者頭部撞擊後初步未達 RED，也可能需要反覆觀察。

所以我們沒有把這些情況藏在一個最終 Tag 裡。Q5–Q12 觸發時，介面會列出風險名稱、級別、原因、時間性和 Prototype 行動建議。大部分 RED 級別的隱藏風險會越過正常的可步行／不可步行分支；Q11 只要求監察，本身不會強制 RED。前線人員因而可同時看到檢傷結果和需要額外小心的原因。

`Unknown` 亦是一個真實答案。信心值公式是 `1 − Unknown 數目 / 12`。Q1、Q5、Q6、Q7 或 Q9 若為 Unknown，信心值最高只會是 `0.4`，並標示要人工覆核；信心值低於 `0.6` 亦會要求覆核。這條公式不是聲稱可以量化所有醫療不確定性，而是避免「沒有資料」在畫面上看起來像「沒有風險」。

程式仍有一個已知輸入驗證問題：Q2／Q3 為 Unknown 時不一定強制覆核，而「呼吸正常但呼吸率為零」這類矛盾資料尚未被硬性拒絕。我們把它寫出來，因為在沒有臨床負責人決定政策前，私自改變檢傷行為並不負責任。

### 為甚麼 AI 只可提高警號，不可降低

有用線索經常留在筆記而非 Checkbox，例如「聲音開始沙啞」、「在爆炸附近但自稱沒事」，或同一觀察以另一種語言寫下。演練用相片也可能顯示表格遺漏的可見線索。這正是 Qwen 可以幫忙的地方。

模型會回傳建議修改的問題及原因。Parser 只保留目前為 `No`、建議改為 `Yes` 的項目。前線人員再選擇接受哪些建議；只有接受後，固定規則才重新計算。

我們採用單向限制，是因為刪除已記錄的危險，比提醒可能遺漏的線索更重大。模型的語言或影像判讀不足以無聲地抹走結構化證據。不過，向上調整的建議也可能錯，所以仍要等人確認。這是一條保守的產品界線，不代表每次 AI 升級在醫療上必然正確。

同一原則也用在其他 AI 功能：

- `qwen3.7-plus` 直接讀多語筆記，避免先翻譯一次而流失意思；
- 演練影像覆核只提出觀察，相片不會進入 LoRa 傷患封包；
- 前線戰術建議先由本機查表，再限制器材、運送級別、PPE 及警戒範圍；Qwen 失效時仍保留查表結果；
- `qwen3.7-max` 接收 Base 的準確即時快照，回傳包含不確定性的結構化策略，但不會重新替傷患分類；
- Base 的 `qwen3.7-plus` Agent 透過六個指定工具讀取現況，不靠舊對話猜數字。

### 為甚麼 Agent 要用工具，而且停在發送之前

Base Agent 可以讀取局勢快照、列出近期傷患、列出目前異常、建立確定性 SITREP、建立 Radio 警報草稿，以及提出廣播請求。它沒有改 Tag 或編輯傷患的工具。每次循環最多六個模型步驟；未知工具名稱或錯誤參數會被拒絕。

使用工具，是因為答案應來自程式當刻的傷患數目，而不是模型記得的舊對話。六步上限，是為了避免操作介面進入無止境的推理循環。把「草擬」與「發送」拆開，則是因為寫出看似合理的文字，不等於有權判斷它正確、必要並適合傳出去。

最後一道界線由程式強制執行。即使模型自行填上 `human_approved=true`，Dispatcher 也會把它改回 false。指揮員必須到 Dashboard 另一部分閱讀草稿，再按下人工核准控制。預設 Broadcaster 目前仍是 Stub，因此 Repository 展示了核准閘門，但沒有聲稱 Base-to-Radio 發送已完成。

### 為甚麼 Radio 資料很小，而且現時要人工貼上

LoRa 適合長距離、低頻寬通訊；Meshtastic 讓多部 Heltec V4 互相接力成網狀網絡，在流動網絡失效時仍可傳送短訊。代價是每一個 byte 都要計算。

EmergencyNet 每名傷患佔 18 bytes，前面另有 10-byte 封包 Header。測試時我們發現一個容易忽略的問題：12 名傷患的 Binary 封包是 226 bytes，但現時 Demo 要以十六進位文字複製；每個 byte 變成兩個字元，結果是 452 characters，超出一般文字路徑。Field 因此每次最多建立四人獨立封包，即 164 個 Hex 字元，其餘傷患留在 Outbox 等下一次發送。

目前完整路徑保留兩次人工複製：

1. Field 產生 Hex，操作員貼到 Meshtastic Android App。
2. Heltec V4 經 LoRa Mesh 傳送及接力。
3. Base 的 Meshtastic Client 收到後，操作員把 Hex 貼到 **Inject test packet**。

我們保留並寫明這條界線，因為它可用現有硬體重現，評審也能逐步檢查。把它稱為「全自動整合」只會掩蓋尚未完成的部分。Direct Binary 傳送、自動匯入 Base、封包身份驗證、防重播及去重，都是下一輪傳輸工程。

### 為甚麼 Base 要跨傷患分析

前線紀錄回答「這名傷患發生甚麼事」；指揮站還要回答「整宗事故正在發生甚麼事」。所以 Base 保留有限的傷患視窗，執行四個固定偵測器：

- 視窗至少有五名傷患後，呼吸急促／微弱或無呼吸者佔至少 50%；
- 視窗至少有三名傷患後，燒傷者佔至少 60%；
- 至少三名受困傷患；
- 十分鐘內收到至少五名 RED 傷患。

門檻全部可見、可測試。Detector 只提出事故層面的警號，不會改任何人的 Tag。之後 Qwen 可解釋即時快照或草擬短警報，由指揮員判斷模式是否成立，以及應調配甚麼資源。

### 為甚麼 Alibaba Cloud 是 Backend 的一部分

Base 已封裝成 Docker Service，可部署到 Alibaba Cloud ECS。Qwen Client 透過 DashScope 相容 OpenAI 的 Endpoint 呼叫 Alibaba Cloud Model Studio。我們以 `qwen3.7-plus` 處理較常用的筆記、影像、戰術及 Tool Calling；只有按需要打開的策略頁使用具 Thinking 的 `qwen3.7-max`。

這種分工令日常路徑較簡單，也讓指揮層得到較強的整合模型。Cloud 的角色因此很具體：ECS 承載可連線的 Base Service，Model Studio 提供額外語言與推理能力；確定性的核心任務仍可在本機完成。

Repository 可證明 Client Code、Container 定義及部署程序，卻**不能**證明現時已有 ECS Instance 正在運行。提交前必須補上真實 Region、公開 Code Link、HTTPS Demo URL 及 Runtime Evidence：**[已核實的 ALIBABA CLOUD 資料]**。

### 一次完整 Demo 應該看到甚麼

好的 Demo 應是一個連續故事，而不是逐個 Tab 參觀：

1. 在平板輸入一名合成的倒塌事故傷患，指出本機規則為甚麼產生該 Tag。
2. 加入多語筆記或演練影像，展示 Qwen 的待確認升級，而不是自動修改。
3. 由前線人員接受或拒絕。
4. 建立四人 Hex 封包、貼進 Meshtastic，並展示至少經一部 Heltec V4 Relay 傳送。
5. 把收到的 Hex 貼到 Base，查看還原後的傷患。
6. 注入第二個已準備封包，令呼吸、燒傷、受困及 RED Surge 偵測器出現。
7. 要求 Agent 回報準確數目並草擬警報。
8. 嘗試由模型提供 Approval，展示它被阻擋；最後才由獨立人工控制決定。

這八步已解釋整個產品：觀察、本機決定、用最少資料傳送、合併報告、使用 Cloud 協助，並把權力留在人手。

### 遇到的難題，以及它們為甚麼重要

從 Repository 可以核實的挑戰，不是戲劇化故事，而是實際設計修正：

1. **小封包不代表文字一定可發送。**只有量度 Hex 而非只看 Binary，才發現必須改成四人 Batch。
2. **Prompt 不是權限邊界。**測試偽造 Approval 後，才確定 Dispatcher 必須強制把模型 Approval 變成 false。
3. **Fallback 必須仍有用。**固定規則和本機查表與模型整理分開，API 失敗時畫面才不會只剩 Error。
4. **更多資料會超出 Radio 預算。**所以相片和長筆記不進傷患封包，只傳精簡訊號。
5. **誠實寫出整合狀態也是安全設計。**人工 Copy、Broadcaster Stub、記憶體儲存及缺少 ECS Proof 都直接列出，不藏在架構箭嘴後面。

團隊實際遇過的硬體故障、Debug 過程、放棄方案、時間壓力及個人分工，必須核實後才加入：**[團隊提供的真實挑戰]**。

### 我們學到甚麼

**可靠是產品功能。**如果網絡一斷，核心流程便消失，再強的模型也沒有用。因此 Tag、Packet、Detector 和 SITREP 都用固定程式處理。

**Human-in-the-loop 必須是一個真正機制。**只寫「人保留控制權」沒有意義。模型要沒有修改工具、建議要等人接受、核准要經另一條 UI 路徑，而且這些條件要有測試。我們按這個標準實作。

**Unknown 必須保持可見。**缺少答案不能靜靜變成 `No`。信心值及覆核提示把資料不完整本身也寫進紀錄。

**真正的傳輸路徑才決定設計。**LoRa 容量、Meshtastic 文字限制及 Hex 膨脹是不同層次；只測第一層，最後會得到一個不能在實際 App 發送的 Demo。

**好的安全文件會寫未完成的部分。**讀者需要知道自動化在哪裡停止、缺少甚麼證據，以及哪些主張仍要由合資格人士審核。

### 下一步

下一個優先事項不是換更大的模型，而是補齊傳輸和證據：

- Meshtastic PortNum Direct Binary 傳送及 Base 自動匯入；
- 真正由人核准的 Outbound Radio Sender；
- 封包身份驗證、防重播、去重、持久化儲存及 Audit Log；
- 多跳距離、延遲、遺失及恢復量度；
- 不同語言及無障礙情境的前線測試；
- 按本地 Protocol 進行受監督的臨床及事故指揮覆核；
- 真實 ECS 部署及可重現的 Runtime Evidence。

設計規則應保持簡單：AI 可以擴闊團隊看到的資訊，但不可抹走已觀察的風險、執行未經覆核的行動，也不可令 Cloud 斷線時整個系統失去作用。
